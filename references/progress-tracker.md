# Progress Tracker Blueprint

Use this blueprint to create `context/progress-tracker.md`.
Replace every bracketed prompt with the current project's state.
Do not copy feature numbers, completed work, architecture decisions, test results, dates, or session notes from previous projects.

```md
# Progress Tracker

Update this file whenever the current phase, active feature, implementation state, assumptions, or handoff notes change.

## Current Phase

- [Project kickoff / Discovery / Planning / Implementation / QA / Stabilization / Launch / Maintenance] — [status]

## Current Goal

- [One sentence describing the active goal for the next session or current work block.]

## Completed

- [Date or session label]: [Completed milestone, document, decision, prototype, feature, or verification.]

## In Progress

- [Active item, owner if relevant, and what remains.]

## Next Up

- [Next concrete item]
- [Next concrete item]

## Open Questions

- [Question blocking product, architecture, design, compliance, or implementation decisions.]

## Assumptions

- [Assumption currently being used so work can continue.]

## Architecture Decisions

- [Decision and short rationale. Link to ADR if one exists.]

## Governance, Compliance, and Review Notes

- [Human review gate, compliance applicability note, permission boundary, or testing requirement.]

## Risks

- [Risk, likelihood/impact if useful, and mitigation or owner.]

## Session Notes

- [Concise note from the latest working session.]

## Verification

- [Command, review, prototype check, user approval, or "not run yet" with reason.]

## Recommended Next Workflow

- [Exactly one recommended next workflow: feature specification, implementation planning, task decomposition, implementation, QA, or user review.]
```
